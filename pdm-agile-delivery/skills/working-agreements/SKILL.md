# Skill: working-agreements

## Purpose
Facilitate creation of explicit team working agreements that govern how the team collaborates, communicates, and makes decisions during delivery.

## When to Use
- Forming a new delivery team
- After team health check reveals process pain
- When joining an existing team as new PDM

## Framework: Working Agreements

### Categories to Cover

**1. Meetings**
- Core hours when everyone is available
- Meeting-free blocks (deep work time)
- Camera-on / camera-optional policy
- Late arrival policy

**2. Communication**
- What goes in Slack vs email vs Jira
- Response time SLAs (Slack: 2 hours; Email: 24 hours)
- How to signal you're in deep work / unavailable
- Escalation path for urgent issues

**3. Delivery Process**
- Definition of Ready (story must be before sprint planning)
- Definition of Done (story must be before marking complete)
- WIP limit agreed by team
- Code review turnaround time (e.g., 24 hours)

**4. Decision Making**
- Who can make which decisions (reference RACI)
- When to escalate vs decide locally
- How disagreements are resolved

**5. Respect & Inclusion**
- How feedback is given (specific, kind, actionable)
- How to disagree constructively
- Psychological safety commitments

### Template Output

**Our Working Agreements — [Team Name] — [Date]**

| Category | Agreement | Agreed By |
|----------|-----------|----------|
| Meetings | Core hours: 10am–4pm local time | ✅ All |
| Communication | Slack response within 2 hours during core hours | ✅ All |
| Delivery | No story moves to Done without passing all acceptance criteria | ✅ All |
| Decisions | Any decision affecting the sprint goal goes to PDM | ✅ All |

Review these agreements every 3 months or after a retrospective flags a process issue.

## Output Format
1. Working agreements document (ready to post in Confluence)
2. Review reminder cadence
3. Suggested next command: `/definition-of-done` or `/wip-limits`
