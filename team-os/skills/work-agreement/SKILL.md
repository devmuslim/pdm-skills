# Work Agreement — ميثاق عمل الفريق

## ما هو Work Agreement؟
وثيقة يتفق عليها الفريق معاً تحدد كيف يعملون، يتواصلون، ويحلون الخلافات.
**ليست قواعد يفرضها المدير — بل اتفاقيات يصنعها الفريق.**

---

## محاور الـ Work Agreement

### 1. 🕐 ساعات العمل والتواصل
```
ساعات العمل الرسمية:    [X] → [Y]
Core Hours (الكل متاح): [X] → [Y]
وقت الاستجابة المتوقع:
  - رسائل Slack/Teams:  خلال [X] ساعات
  - Emails:             خلال [X] يوم عمل
  - Urgent (P1):        خلال [X] دقيقة
هل العمل عن بُعد مسموح؟ نعم / لا / هجين
أيام WFH المسموح بها:   [X]
```

### 2. 📅 الاجتماعات
```
الحضور:        إلزامي لـ Ceremonies / اختياري لـ [X]
التأخير:       مقبول حتى [X] دقائق — بعدها يُبدأ بدون
الكاميرا:      [إلزامية / مفضلة / اختيارية]
التسجيل:       [نعم / لا / بإذن]
لا اجتماعات:   [يوم X] — يوم تركيز عميق
إلغاء الاجتماع: إشعار قبل [X] ساعة على الأقل
```

### 3. 💻 قواعد الكود والعمل التقني
```
Branch Naming:     feature/[ticket-id]-short-description
Commit Messages:   [Conventional Commits / Free / غيره]
Code Review:       كل PR يحتاج [X] Approvals
Review Time:       PR يُراجع خلال [X] ساعة عمل
Definition of Done: [رابط لـ DoD المتفق عليها]
لا Merge:          بدون Tests خضراء + Approval
```

### 4. 🤝 التعاون وحل الخلافات
```
إذا اختلفنا على القرار التقني:
  1. كل طرف يعرض رأيه بالأدلة (5 دقائق)
  2. نصوّت — الأغلبية تقرر
  3. إذا تعادلنا — [Tech Lead / SM / PO] يقرر
  4. الجميع يلتزم بعد القرار

إذا واجهت مشكلة شخصية مع زميل:
  1. تكلّم معه مباشرة أولاً
  2. إذا لم يُحل — اجلب Scrum Master
  3. كل شيء يُعالج باحترام وسرية

Psychological Safety:
  ✅ لا يوجد سؤال غبي
  ✅ الأخطاء فرص للتعلم لا للعقاب
  ✅ الاختلاف في الرأي مرحب به
```

### 5. 📢 مشاركة المعرفة
```
Documentation:  كل قرار تقني يُوثَّق في [Confluence / Wiki]
Learning:       [X] ساعة أسبوعياً للتعلم الذاتي
Knowledge Share: جلسة واحدة شهرياً — كل عضو يعلّم شيئاً
Pair Programming: مسموح ومشجَّع — خاصة للمهام الصعبة
```

### 6. 🚨 الطوارئ والغياب
```
إذا غبت:
  1. أخبر الفريق في [Channel] قبل [X] ساعات
  2. سلّم مهامك لـ [شخص محدد أو SM]
  3. حدّث Jira بحالة مهامك

Escalation Path:
  مشكلة تقنية    → Tech Lead
  مشكلة نطاق     → Product Owner
  عائق خارجي     → Scrum Master
  أزمة إنتاج     → [خط طوارئ محدد]
```

### 7. 🔄 مراجعة الاتفاقية
```
تُراجع: كل [3 أشهر] في Retrospective
يمكن لأي عضو اقتراح تعديل في أي وقت
التعديل يحتاج موافقة [أغلبية / الجميع]
التاريخ الأخير للمراجعة: [X]
```

## نصائح للـ Agile Coach عند بناء Work Agreement

```
✅ اجعل الفريق يكتبها معاً — لا تمليها عليهم
✅ ابدأ بـ Liberating Structure: "1-2-4-All"
   - كل شخص يكتب 3 قواعد يريدها (1 دقيقة)
   - ناقش مع زميل (2 دقائق)
   - ناقش في مجموعة 4 (4 دقائق)
   - شارك مع الكل وصوّتوا
✅ اجعلها مرئية (على الجدار أو Confluence)
✅ راجعها في Retro الأولى بعد شهر
❌ لا تجعلها طويلة جداً — 1-2 صفحات كافية
```
